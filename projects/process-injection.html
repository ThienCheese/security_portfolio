Red-team scenario

How the loader bypass chain was designed

Objective

Build a custom loader that can reduce Defender visibility by avoiding obvious static indicators, bypassing user-mode API hooks, disabling selected user-mode telemetry paths, and executing a payload inside a trusted process.

Result demonstrated

The full chain succeeded in the Windows 11 VM as well as a production environment desktop with Defender real-time protection enabled, while still leaving runtime artifacts that a custom detector could use.

Experiment timeline

From blocked injection to successful evasion

Test 1

Classic Windows API injection

Baseline attempt using the common pattern of allocating memory, writing payload, changing permissions, and creating a remote thread. Defender blocked this path in the lab.

Test 2

Direct syscall without full telemetry impairment

User-mode hook visibility was reduced, but kernel-level and heuristic signals still provided enough detection opportunity.

Test 3

Full evasion chain

Anti-sandbox, dynamic resolution, telemetry impairment, payload encryption, and injection were combined. The payload executed successfully in the lab environment.

Blue

Detector validation

The same attack still left runtime traces: a new thread and executable private memory inside the target process.

Detector architecture

Turning offensive understanding into detection

The detector does not try to catch the API call itself. It inspects the aftermath of injection: thread start address and memory region type.

Engine 1

ETW Kernel Trace

Observes process events to provide context around parent-child relationships and suspicious process activity.

Engine 2

Thread Polling

Compares thread snapshots, detects new threads in sensitive processes, and checks whether the start address points to suspicious executable memory.

Engine 3

Memory Scanner

Scans target processes for committed private executable regions that are not backed by a legitimate image file.

Detection flowchart

How the tool reasons about injection

01

Build baseline

Collect known process and thread state.

02

Watch for new thread

Identify thread IDs that appear after baseline.

03

Query start address

Inspect where the new thread begins executing.

04

Inspect memory type

Check whether the address belongs to a private executable region.

05

Confirm with memory scan

Search for MEM_PRIVATE + executable pages in sensitive targets.

06

Alert / block

Raise a high-confidence T1055 signal and optionally terminate in the lab.

Why MEM_PRIVATE matters

MEM_IMAGENormal code loaded from EXE/DLL
MEM_PRIVATE + EXECUTESuspicious for injected shellcode
MEM_PRIVATE + READ/WRITECommon data/heap area

Limitations and next steps

Current detection focuses on private executable memory and classic remote thread patterns. Future work should cover module stomping, process hollowing, APC injection, call stack analysis, and event-driven monitoring for lower latency.
