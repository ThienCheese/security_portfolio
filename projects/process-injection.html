Red-team research model

How the loader bypass chain was designed

The public page explains the research architecture and learning outcomes, not a step-by-step payload guide.

Objective

Build a controlled loader that reduces Defender visibility by avoiding static indicators, resolving sensitive functions dynamically, bypassing user-mode hook assumptions, impairing selected telemetry paths, decrypting payload content at runtime, and injecting into a trusted process.

Result demonstrated

The full chain succeeded in the lab while still leaving runtime artifacts: new thread creation and private executable memory inside the target process. Those artifacts became the basis for the custom detector.

Technique stack

What each technique shows

01

Anti-sandbox delay

Measures timing behavior to avoid running the chain when sleep is skipped or emulation is abnormal.

02

XOR string hiding

Decrypts sensitive strings at runtime to reduce plain-text indicators and simple static rules.

03

Rolling XOR payload

Stores payload data encrypted and decrypts it only in memory immediately before execution.

04

SSN extraction

Reads the NT function prologue to identify the System Service Number needed for direct syscall stubs.

05

Direct syscall stubs

Creates small in-memory syscall dispatchers to reduce reliance on user-mode hooked API paths.

06

AMSI / ETW impairment

Demonstrates why defenders should not rely on a single telemetry channel or hook layer.

07

Trusted process target

Locates a common user process and injects into it to reduce the visibility of a standalone suspicious process.

08

Remote thread execution

Uses the classic final artifact of process injection: a new thread begins execution from injected memory.

09

Blue-team translation

Turns offensive research into detector logic focused on memory type and thread start address.

Experiment timeline

From blocked injection to behavioral detection

Test 1

Classic Windows API injection

Baseline pattern: allocate memory, write payload, change protection, create remote thread. This path was easier for Defender to block or flag.

Test 2

Native NT API and syscall path

The loader reduced user-mode API visibility by resolving NT functions and dispatching through direct syscall concepts.

Test 3

Full evasion chain

Anti-sandbox checks, string/payload encryption, telemetry impairment, and process injection were combined to achieve execution in the lab.

Blue

Detector validation

The attack still left behavioral traces: a new thread and executable private memory inside a trusted target process.

Detector architecture

Turning offensive knowledge into detection

The detector does not try to catch the API call itself. It inspects the aftermath of injection: process/thread changes, thread start address, and memory region type.

Engine 1

ETW Kernel Trace

Observes process events to add context around parent-child relationships and suspicious activity in high-value processes.

Engine 2

Thread Polling

Builds a thread baseline, detects new threads in existing processes, queries their start address, and checks whether they begin inside suspicious executable memory.

Engine 3

Memory Scanner

Scans target processes for committed private executable regions that are not backed by normal image files.

Detection logic

Why MEM_PRIVATE matters

Memory reasoning

MEM_IMAGENormal code loaded from EXE/DLL
MEM_PRIVATE + EXECUTESuspicious for injected shellcode
MEM_PRIVATE + READ/WRITECommon data/heap area

Detection conditions

  • New thread appears in an existing target process.
  • Thread start address points to a committed private region.
  • Memory protection is executable, especially RWX or large executable private regions.
  • Target process is high-value or commonly abused, such as explorer.exe, svchost.exe, browser, shell, or security-sensitive processes.
