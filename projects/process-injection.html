Detector architecture

Three complementary engines

Engine 1

ETW Kernel Trace

Observes process events to provide context around parent-child relationships and suspicious process activity.

Engine 2

Thread Polling

Compares thread snapshots, detects new threads in sensitive processes, and checks whether the start address points to suspicious executable memory.

Engine 3

Memory Scanner

Scans target processes for committed private executable regions that are not backed by a legitimate image file.

Detection rule of thumb

Why MEM_PRIVATE matters

MEM_IMAGENormal code loaded from EXE/DLL

MEM_PRIVATE + READ/WRITECommon data/heap area

The detector does not rely on API hook visibility. It looks for runtime artifacts left after injection: new thread start address and executable private memory.

Experiment summary

  • Traditional API-based injection was blocked by default protection.
  • Direct syscall concepts reduced user-mode hook visibility.
  • Full evasion chain succeeded in the lab, but still left memory/thread artifacts.
  • The custom detector identified the injected process using runtime behavior.

Limitations and next steps

Current detection focuses on private executable memory and classic remote thread patterns. Future work should cover module stomping, process hollowing, APC injection, call stack analysis, and event-driven monitoring for lower latency.
