Self-Hosted DNS Firewall & Secure DNS Gateway

A containerized DNS security platform that filters advertising, tracking, phishing, and malicious domains for an entire local network. The project combines a Python DNS service, automated threat-list updates, DNS sinkholing, query analytics, Caddy, Docker Compose, and Cloudflare Tunnel for encrypted remote access through DNS-over-HTTPS.

Architecture

One filtering engine, two access paths

Local devices resolve through the server on port 53, while remote devices use an HTTPS endpoint exposed through Cloudflare Tunnel. Both paths reach the same filtering, forwarding, logging, and policy layer.

LAN path

Automatic protection for every client

The Ubuntu host uses a static LAN address. The router advertises that address as the primary DNS server through DHCP, allowing connected devices to use the firewall without installing individual agents.

Remote path

Encrypted access without public exposure

Cloudflared creates an outbound tunnel to Cloudflare. Remote clients send DNS-over-HTTPS requests to the project domain, avoiding static public IP requirements and inbound router port forwarding.

Request lifecycle

How one DNS query is evaluated

01

Receive

Accept a LAN DNS request or a remote DoH request forwarded by Caddy.

02

Normalize

Extract and normalize the queried domain before policy evaluation.

03

Evaluate

Compare the domain against local and synchronized blacklist entries.

04

Resolve

Return a sinkhole address for blocked domains or forward allowed queries upstream.

05

Record

Store query status and expose totals, blocked rate, and domain trends in the dashboard.

Security capabilities

What the platform provides

01 · DNS policy enforcement

Blacklist-based filtering

Blocks domains associated with advertisements, trackers, phishing, malware, and unwanted content before application traffic is established.

02 · Threat prevention

DNS sinkholing

Returns a controlled local address for blocked domains, interrupting resolution and reducing exposure to known malicious infrastructure.

03 · Privacy

DNS-over-HTTPS

Protects remote DNS traffic from casual interception and manipulation while clients are outside the trusted home network.

04 · Operations

Automated list updates

Synchronizes block sources every 24 hours to keep filtering policy current without requiring manual rule maintenance.

05 · Visibility

Query logging and dashboard

Tracks total queries, blocked requests, block rate, frequently queried domains, and frequently blocked domains.

06 · Deployment

Containerized services

Separates DNS, proxy, and tunnel components with Docker Compose for repeatable deployment, troubleshooting, and recovery.

Implementation

Technology responsibilities

Python

DNS processing engine

Receives DNS requests, applies filtering logic, produces sinkhole responses, forwards allowed domains, and records query results.

Docker Compose

Service orchestration

Builds and operates the DNS service, Caddy, and cloudflared as an integrated but independently observable stack.

Caddy

HTTPS gateway

Routes the dashboard and /dns-query endpoint while protecting administration with authenticated access.

Cloudflare Tunnel

Remote connectivity

Publishes the web and DoH service through an outbound encrypted connection, including support for CGNAT environments.

Netplan + Router DHCP

LAN integration

Provides a stable server address and distributes the DNS firewall automatically to clients connected to the network.

Threat lists

Policy sources

Supports synchronized sources such as StevenBlack, OISD, and Hagezi, plus manually managed blacklist entries.

Deployment workflow

From clean Ubuntu host to working gateway

Step 01

Prepare the Linux server

Deploy Ubuntu or Debian, configure bridged networking where applicable, assign a static address through Netplan, and validate gateway connectivity.

Step 02

Deploy the containers

Install Docker and Docker Compose, configure environment variables, create the Caddy password hash, and start the stack with a reproducible Compose workflow.

Step 03

Connect Cloudflare

Create a Zero Trust tunnel and route the public hostname to Caddy so the dashboard and DoH endpoint are accessible without exposing the home router.

Step 04

Integrate the LAN

Configure the router DHCP DNS option to point clients to the local DNS server and renew client leases.

Step 05

Validate and benchmark

Use nslookup, dig, curl, or kdig to verify allowed and blocked responses; use dnsperf for performance testing.

Validation

Tests

Functional checks

  • Allowed lookup: query a legitimate domain against the local server and confirm a valid upstream response.
  • Blocked lookup: query a blacklisted advertising domain and confirm the sinkhole address is returned.
  • Remote DoH: send a request to the public /dns-query endpoint and validate the DNS response.
  • Client assignment: verify that DHCP clients receive the firewall address as their resolver.
  • Operational status: inspect container health and service logs during troubleshooting.
